<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser localStorage → App | JWT token stored in localStorage is readable by any JS running on the origin. XSS can exfiltrate it. |
| App → Backend (via Vite proxy) | Login/register credentials sent over HTTP in development (localhost). JWT sent in Authorization header on every authenticated request. |
| Backend → App | Backend returns JWT token and user data. Response shapes (id vs _id, { user } wrapping) must be handled correctly. |
| User input → Form | Email, password, name, bio fields cross the trust boundary from untrusted user input to the app. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02 | Information Disclosure | localStorage JWT storage | accept | JWT in localStorage is the standard SPA pattern. XSS risk exists but React's JSX auto-encoding prevents most injection vectors. HTTP-only cookies would require backend changes (out of scope). CSP headers recommended for production deployment. |
| T-01-03 | Tampering | CSRF on state-changing requests | accept | JWT is sent via `Authorization: Bearer` header, not cookies. Browsers do not auto-attach this header, making CSRF attacks ineffective against this SPA. |
| T-01-04 | Information Disclosure | Credentials over HTTP (dev) | accept | Both frontend (Vite port 3000) and backend (Express port 4000) run on localhost in development. No network exposure. Production requires HTTPS — documented in deployment guide (out of Phase 1 scope). |
| T-01-05 | Information Disclosure | Token leakage via browser extensions | accept | Inherent localStorage risk. Mitigation requires httpOnly cookies + backend session management (out of scope). Token is opaque JWT — server determines authorization. |
| T-01-06 | Information Disclosure | User enumeration via login responses | mitigate | Backend returns generic "Invalid email or password" for both wrong-email and wrong-password cases (confirmed in `authController.ts` line 69). Frontend displays this message verbatim — Login.tsx does NOT differentiate "email not found" vs "wrong password." |
| T-01-07 | Elevation of Privilege | 401 not propagating across tabs | accept | If a user logs out in one tab, other tabs retain the token in memory until next API call triggers the 401 interceptor. Full tab-sync via `storage` event listener is a nice-to-have (Phase 4 polish). Current behavior: token is cleared from localStorage on logout; other tabs will 401 on next request and redirect. |
| T-01-08 | Information Disclosure | XSS via form input rendering | mitigate | React JSX auto-escapes all string content rendered in JSX. Form input values are strings bound to state, not rendered as HTML. Backend error messages (`err.response?.data?.message`) are rendered as text content in JSX (`<p>{error}</p>`) — React escapes them. Zod validation messages are static strings defined in schemas. |
| T-01-09 | Spoofing | JWT token forgery | accept | Token signing and verification happens on the backend (HS256, `JWT_SECRET` from `backend/.env`). Frontend only stores and transmits the token. If the backend verifies a token as valid, the frontend treats it as valid. This is correct — the backend is the authoritative verifier. |
| T-01-10 | Repudiation | No audit log for auth actions | accept | Phase 1 does not include audit logging. Login/logout events are transient client-side actions. Out of scope for walking skeleton. |
| T-01-11 | Elevation of Privilege | ProtectedRoute bypass via React DevTools | accept | Client-side route guards are a UX convenience, not a security boundary. The real access control is on the backend (`protect` middleware in `auth.middleware.ts`). An attacker who modifies React state can see protected UI shells but cannot fetch data from protected endpoints (401 from backend). |
</threat_model>
