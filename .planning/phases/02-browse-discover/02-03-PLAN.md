## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Recipe detail data fetched from backend (GET /api/recipes/:id) |
| client→API | Comments posted by authenticated users (POST /api/recipes/:id/comments) |
| client→API | Comments deleted by authorized users (DELETE /api/comments/:id) |
| client→API | Recipe deleted by author (DELETE /api/recipes/:id) |
| user→UI | Comment content rendered in the DOM (XSS vector) |
| user→UI | Ingredient checklist state (local only, no server trust boundary) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|----------------|
| T-02-10 | Spoofing | RecipeDetail.tsx comment post | mitigate | Comment POST requires valid JWT token (axios interceptor attaches `Authorization: Bearer` header). 401 interceptor redirects to /login on invalid/expired token. Comment is attributed to the authenticated user server-side (userId from token, not request body). |
| T-02-11 | Tampering | RecipeDetail.tsx comment delete | mitigate | Delete button only rendered when `comment.userId === user._id` (client-side check). Backend additionally verifies ownership via JWT. Window.confirm() provides second-factor user intent confirmation before DELETE request. |
| T-02-12 | Tampering | RecipeDetail.tsx recipe delete | mitigate | Delete controls only visible when `recipe.authorId === user._id`. Inline confirmation box ("Confirm Deletion?") with explicit Confirm/Cancel buttons prevents accidental deletion. Backend verifies author ownership via JWT. |
| T-02-13 | Information Disclosure | RecipeDetail.tsx comment content rendering | mitigate | Comment content is rendered via React JSX (`{comment.content}` in a `<p>` tag) — React automatically escapes HTML entities, preventing XSS. No `dangerouslySetInnerHTML` used. |
| T-02-14 | Repudiation | RecipeDetail.tsx comment/recipe actions | accept | Backend is responsible for audit logging. Frontend displays success/failure via sonner toasts for user confirmation. No client-side audit trail needed for ASVS L1. |
| T-02-15 | Denial of Service | RecipeDetail.tsx comment form | mitigate | Comment text validation: empty submission blocked client-side (`!newCommentText.trim()`). Submit button disabled during submission (`submittingComment` state, `disabled` prop, `cursor-wait`). Rating bounded 1-5 (StarRating component only emits values in range). |
| T-02-16 | Elevation of Privilege | RecipeDetail.tsx comment post unauthenticated | mitigate | Comment form is conditionally rendered: authenticated users (`token` truthy) see the form, unauthenticated users see "Sign in to leave a comment" CTA linking to /login. No form bypass possible from UI. |
| T-02-17 | Information Disclosure | RecipeDetail.tsx ingredient checklist | accept | Ingredient check state is local (useState<Set<number>> in memory). No data sent to backend. No cross-session persistence. Resets on component unmount — no information retained. |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | `npx shadcn@latest add textarea checkbox separator` uses official shadcn registry (verified in 02-UI-SPEC.md Registry Safety) |
</threat_model>
