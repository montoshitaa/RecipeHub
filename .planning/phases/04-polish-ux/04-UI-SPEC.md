## Spacing Scale

Declared values from Tailwind v4 default 8-point scale (confirmed in use throughout codebase):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding between icon+text |
| sm | 8px | Compact element spacing, gap-2 |
| md | 16px | Default element spacing, card padding (p-4), gap-4 |
| lg | 24px | Section padding (p-6), grid gaps (gap-6) |
| xl | 32px | Layout gaps (gap-8), major padding (p-8) |
| 2xl | 48px | Major section breaks (space-y-12) |
| 3xl | 64px | Page-level vertical spacing (space-y-16, py-12) |

### Standardized Responsive Padding Contract

| Context | Mobile | Tablet (sm: 640px+) | Desktop (lg: 1024px+) |
|---------|--------|---------------------|----------------------|
| Page sections | `p-4` or `p-5` | `sm:p-6` or `sm:p-8` | `lg:p-8` |
| Card padding | `p-4` | same | same |
| Hero/editorial blocks | `p-6` | `sm:p-10` | same |
| Form containers | `p-6` | `sm:p-8` | same |
| Grid gaps | `gap-6` | `sm:gap-8` | same |
| Section margins | `space-y-8` | same | same |

Exceptions: Auth forms (Login, Register) use `max-w-[400px]` centered with `p-6 sm:p-8` — this is intentional for narrow auth UI and matches design convention. Icon-only touch targets on mobile (delete/edit buttons, checkboxes) must be at least 44px — achieved via padding on clickable elements (`p-2` on icon buttons = 44px minimum).

## Typography

### Type Scale

| Role | Size | Weight | Line Height | Font Family | Usage |
|------|------|--------|-------------|-------------|-------|
| Body | 15px (`text-[15px]`) | 400 (regular) | 1.6 | font-sans (IBM Plex Sans) | Paragraphs, descriptions, form inputs, comment text |
| Body (large) | 17px (`text-[17px]`) | 400 (regular) | 1.6 | font-sans | Preparation steps body text |
| Body (small) | 14px (`text-sm`) | 400 (regular) | 1.5 | font-sans | Comment content, secondary descriptions |
| Label | 11px (`text-[11px]`) | 700 (bold) | 1.4 | font-mono (IBM Plex Mono) | Form labels, filter labels (uppercase, tracking-wider) |
| Meta | 10-13px (`text-[10px]`-`text-[13px]`) | 500-700 | 1.3 | font-mono | Category badges, timestamps, "Member since", meta rows |
| Heading (section) | 20-24px (`text-xl`-`text-2xl`) | 700 (bold) | 1.2 | font-serif (IBM Plex Serif) | Section titles (Profile, Comments, Ingredients) |
| Heading (page) | 36-48px (`text-4xl sm:text-5xl`) | 700 (bold) | 1.1 | font-serif | Page titles (RecipeDetail, Home hero) |
| Display | 36-48px (`text-4xl sm:text-5xl`) | 700 (bold) | 1.0 (leading-none) | font-serif | Recipe title in detail view |

### Font Weight Contract

Only two weights are used across the app. No thin, light, medium, or extrabold.

| Weight | Token | Usage |
|--------|-------|-------|
| 400 | Regular | Body text, descriptions, input placeholders, comment content |
| 700 | Bold | All headings, labels, buttons, meta labels, navigation links, badge text |

### Typography Conventions

- **Headings**: Always `font-serif` (IBM Plex Serif) + `font-bold` + `tracking-tight`. Never use font-mono or font-sans for titles.
- **Labels**: Always `font-mono` (IBM Plex Mono) + `uppercase` + `tracking-wider` + `font-bold`. Size is `text-[11px]`. Used for form field labels, filter labels, meta section headers.
- **Buttons**: Always `font-mono` + `uppercase` + `tracking-wider` + `font-bold`. Size is `text-xs` or `text-[11px]`.
- **Meta/ancillary**: Always `font-mono`. Used for dates, counts, category names, difficulty text in cards. Uppercase at `text-[10px]-text-[11px]` with `tracking-wider`.
- **Body paragraphs**: Always `font-sans` + `leading-relaxed`. Size is `text-sm` (14px), `text-[15px]`, or `text-[17px]` (steps only).

## Color

### 60/30/10 Split

| Role | CSS Variable | Hex Value | Usage |
|------|-------------|-----------|-------|
| Dominant (60%) | `--color-bg` | `#f5f4f0` | Page background — all pages, all states. Warm off-white/cream. |
| Secondary (30%) | `--color-surface` | `#ffffff` | Cards, form containers, header, filter bars, comment boxes, meta bars, ingredient table containers. Pure white for contrast against bg. |
| Accent (10%) | `--color-accent` | `#c0392b` | Primary CTAs, difficulty badge "Hard", recipe hero placeholder initials background, link hover color, left decorative border on recipe description |

### Accent Reserved For

The accent color `#c0392b` (deep wine red) is STRICTLY reserved for:
1. **Primary CTA buttons**: "Clear Search & Filters" (Home), "Create my first recipe" (Profile), "Sign in to leave a comment" (RecipeDetail)
2. **Difficulty badge "Hard"**: Used in `RecipeCard` and recipe detail meta
3. **Recipe hero placeholder**: Initial letter background when no image (RecipeDetail hero, RecipeCard placeholder)
4. **Link hover state**: `hover:text-accent` on navigation links, card titles, and interactive text
5. **Decorative accents**: Left border on recipe description paragraph (`border-l-2 border-accent`), star rating number badge background

Accent MUST NOT be used for: form submit buttons (use `bg-text-custom`), navigation (use `text-text-custom`), informational badges, or any non-CTA element.

## Responsive Breakpoint Contract

### Breakpoint Definitions

From Tailwind v4 defaults (verified in use):

| Prefix | Min Width | Typical Device | Grid Columns | Layout Mode |
|--------|-----------|----------------|-------------|-------------|
| *(unprefixed)* | 0px | Mobile (375px+) | 1 column | Stacked (flex-col) |
| `sm:` | 640px | Large phone / small tablet | 2 columns (cards) | Row where appropriate (flex-row) |
| `md:` | 768px | Tablet portrait | 2 columns (cards) | Row + intermediate layouts |
| `lg:` | 1024px | Tablet landscape / desktop | 3 columns (cards), 6/4 split (detail) | Full desktop layouts |
| `xl:` | 1280px | Desktop | 3 columns max | Max-width containers |

### Page-by-Page Responsive Contract

| View | Mobile (0-639px) | Tablet (640-1023px) | Desktop (1024px+) |
|------|-----------------|---------------------|-------------------|
| Home (feed grid) | `grid-cols-1` | `md:grid-cols-2` | `lg:grid-cols-3` |
| Home (hero block) | `flex-col` | `md:flex-row` | `md:flex-row` |
| Home (filter bar) | `flex-col`, inputs full-width | `flex-col`, inputs full-width | `lg:flex-row` |
| RecipeDetail (main layout) | Single column | Single column (max readability) | `lg:grid-cols-10` (6:4 split) |
| RecipeDetail (meta bar) | `grid-cols-2` | `md:grid-cols-4` | `md:grid-cols-4` |
| RecipeDetail (comments) | Single column | Single column | `lg:grid-cols-10` (6:4 split) |
| RecipeCard | Full width in grid cell | Full width in grid cell | Full width in grid cell |
| RecipeForm (ingredients) | Card layout (mobile) | Table layout (`hidden sm:block`) | Table layout |
| RecipeForm (category/difficulty) | `grid-cols-1` | `md:grid-cols-2` | `md:grid-cols-2` |
| RecipeForm (time/servings) | `grid-cols-1` | `md:grid-cols-2` | `md:grid-cols-2` |
| RecipeForm (footer buttons) | `flex-col`, `w-full` | `sm:flex-row`, `sm:w-auto` | `sm:flex-row`, `sm:w-auto` |
| Login/Register | Centered, `max-w-[400px]` | Same | Same |
| Profile (user card) | `flex-col`, `text-center` | `sm:flex-row`, `sm:text-left` | `sm:flex-row` |
| Profile (recipe grid) | `grid-cols-1` | `md:grid-cols-2` | `lg:grid-cols-3` |
| Header | `flex-col`, `flex-wrap` nav | `sm:flex-row` | `sm:flex-row` |
| Footer | `flex-col`, `text-center` | `md:flex-row`, `md:text-left` | `md:flex-row` |

### Responsive Design Rules

1. **Mobile-first always**: Unprefixed classes = mobile default. Add `sm:`, `md:`, `lg:` overrides only when needed.
2. **No fixed pixel widths**: Never use `w-[400px]` without `max-w-full` protection. Auth forms intentionally use `max-w-[400px]` (narrow by design).
3. **Images**: Always use `aspect-[16/9]` with `object-cover`. Never set explicit height in px — use aspect ratio.
4. **Grid max columns**: 3 columns MAX at `lg:`. Do not add `xl:grid-cols-4` — recipe cards become too narrow.
5. **No horizontal overflow**: Verify every view at 375px width. Any overflowing element needs `max-w-full` or `overflow-x-auto`.
6. **Touch targets**: Icon-only buttons (`p-2`) = 44px minimum on mobile. Checkboxes in ingredient rows must be tappable.
7. **Tablet intermediate**: Never skip from mobile directly to `lg:`. Always provide `md:` (or `sm:`) intermediate layout.

## Loading State Pattern Contract

### Pattern Catalog

**Pattern A — Skeleton Grid (card feeds)**
- Used by: Home feed, Profile recipe grid
- Structure: Same grid as content (`grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6 sm:gap-8`)
- Card skeleton: `<Skeleton>` for image area + `<Skeleton>` bars for title/meta
- Card count: 3 placeholder cards (not dynamically matching content count)
- Wrapper: `border border-border-custom bg-surface` — identical to real card
- Animation: Handled by `<Skeleton>` internal `animate-pulse`

**Pattern B — Full-Page Skeleton (detail/edit views)**
- Used by: RecipeDetail, EditRecipe
- Structure: Mirrors real two-column layout with `<Skeleton>` blocks
- Image placeholder: `<Skeleton className="w-full aspect-[16/9]">` inside `border border-border-custom bg-surface`
- Text placeholders: `<Skeleton className="h-4 w-3/4">`, `<Skeleton className="h-10 w-1/2">`
- Wrapper: `space-y-12 animate-pulse` for page-level; individual sections use container borders

**Pattern C — Button Spinner (form submissions)**
- Used by: Login, Register, CreateRecipe, EditRecipe, comment submit
- Structure: `<Button disabled={isSubmitting}>` wrapping `<Spinner data-icon="inline-start" />` + loading text
- Spinner: `<Spinner className="size-4" />` from `@/components/ui/spinner`
- Button: `disabled:opacity-50 disabled:cursor-wait` prevents double-submit

**Pattern D — Inline Progress Text (legacy, being replaced)**
- Current: Text-only "Publishing recipe...", "Authenticating...", "Submitting feedback..."
- Target: Replace all with Pattern C (Spinner + text). Pattern D is DEPRECATED.

## Empty State Pattern Contract

### shadcn Empty Component Structure

The shadcn `<Empty>` component uses a composition API. The standardized structure for this phase:

```tsx
import { Empty, EmptyContent, EmptyDescription, EmptyHeader, EmptyMedia, EmptyTitle } from "@/components/ui/empty"
```

| Slot | Required | Purpose |
|------|----------|---------|
| `<Empty>` | Yes | Root wrapper |
| `<EmptyHeader>` | Yes | Groups title + description + media |
| `<EmptyMedia variant="icon">` | Yes | Icon container — use lucide-react icons |
| `<EmptyTitle>` | Yes | Bold heading |
| `<EmptyDescription>` | Yes | Descriptive text |
| `<EmptyContent>` | Only if CTA present | Action area (button or link) |

### Empty State Icons

| View | Icon | lucide Import |
|------|------|---------------|
| Home (no search results) | Search | `Search` from `lucide-react` |
| Profile (no recipes) | Utensils | `Utensils` from `lucide-react` |
| RecipeDetail (no comments) | MessageSquare | `MessageSquare` from `lucide-react` |

### When NOT to Use Empty Component

Do NOT use the full `<Empty>` composition for:
- Form-only views (Login, Register, CreateRecipe, EditRecipe) — these never show empty data
- Static pages (NotFound, Unauthorized) — these are error states, not empty states
- Sub-section minor empty states within a page that already has content — retain inline text (e.g., "No reviews yet" in RecipeDetail comments section, if the recipe content above is already displayed)

## Error State Pattern Contract

### Pattern: Inline Error Banner

```tsx
{error && !loading && (
  <div className="bg-red-50 border border-danger text-danger p-4 sm:p-5 font-mono text-sm">
    <p className="font-bold uppercase tracking-wider mb-2">ERROR HEADING</p>
    <p>{error}</p>
    <button onClick={retryFn} className="mt-3 text-xs uppercase tracking-wider underline hover:no-underline font-bold">
      Try again
    </button>
  </div>
)}
```

### Error State Rules

1. **Always show retry** for data-fetching views: Home, Profile. The retry button must call the same fetch function.
2. **Show return navigation** for detail views that can't load: RecipeDetail ("Return to feed"), EditRecipe ("Return to feed").
3. **Never show retry** for form submission errors (Login, Register, CreateRecipe) — the form remains usable; user can correct and resubmit.
4. **Error banners replace the content area** — do not show both error banner and skeleton/empty state.
5. **React auto-escapes** all `{error}` expressions — no XSS risk. Never use `dangerouslySetInnerHTML`.
6. **Toasts complement banners** for mutation failures (create/edit/delete/comment). Inline banners for data-fetching views; toasts for form submissions.

## State Rendering Priority

When a view has data-fetching, the rendering priority is:

```
1. loading === true        → Show loading state (Skeleton or Spinner)
2. error !== null          → Show error banner (with retry/return CTA)
3. data.length === 0       → Show empty state (with CTA if applicable)
4. otherwise               → Show content
```

This is the existing pattern used in Home.tsx and RecipeDetail.tsx. All views MUST follow this order.

## Additional Interaction Contracts

### Toast Notifications (sonner)

| Trigger | Type | Message |
|---------|------|---------|
| Login success | `toast.success` | "Welcome back!" |
| Register success | `toast.success` | "Account created successfully!" |
| Recipe created | `toast.success` | "Recipe published successfully!" |
| Recipe updated | `toast.success` | "Recipe updated successfully!" |
| Recipe deleted | `toast.success` | "Recipe deleted" |
| Comment posted | `toast.success` | "Comment posted" |
| Comment deleted | `toast.success` | "Comment deleted" |
| Any mutation failure | `toast.error` | API error message or fallback (see Copywriting Contract) |

### Form Validation Errors

Inline field errors display below the field in `text-danger text-[11px] font-mono`. They appear on:
- Blur (react-hook-form default mode)
- Submit attempt (react-hook-form validates all fields)

Form-level errors (API errors not tied to a specific field) display as a banner at the top of the form (see Error State Pattern Contract).

### Button States

| State | Visual |
|-------|--------|
| Default | Full opacity, interactive, cursor-pointer |
| Disabled (submitting) | `disabled:opacity-50 disabled:cursor-wait` |
| Hover | See component-specific hover styles (e.g., `hover:bg-white hover:text-text-custom` on dark buttons) |
| Focus | `focus:outline-none focus:border-text-custom focus:bg-white` (inputs) or default browser ring (buttons) |

### Ingredient Checklist (RecipeDetail)

Checkbox interaction is local state only — no API calls. Checked items get:
- `line-through decoration-border-custom/50 text-text-muted/60` on both ingredient name and amount
- State resets on page leave (stored in `useState<Set<number>>`)
